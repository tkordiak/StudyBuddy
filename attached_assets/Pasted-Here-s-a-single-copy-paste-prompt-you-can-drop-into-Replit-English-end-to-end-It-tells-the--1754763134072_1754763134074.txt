* **Deployment:** Replit web server + environment variables.

### 2) Core user flow (Happy path)

### 4) Data model (Prisma)

```prisma

  email        String   @unique

model Generation {
  id           String   @id @default(cuid())
  userId       String
  createdAt    DateTime @default(now())
  company      String
  role         String
  jobUrl       String?
  resumeInput  String   // normalized text
  jdExtract    String?  // scraped JD if available
  tailoredResumeMd String
  coverLetterMd    String
  tokensUsed    Int     @default(0)

model Upload {
  id        String   @id @default(cuid())
  userId    String
  filename  String
  text      String
  createdAt DateTime @default(now())
  User      User     @relation(fields: [userId], references: [id])
}
```

### 5) Backend API (Express, TypeScript)

* `POST /api/auth/request-code` ‚Üí send OTP to email (log to console in dev).
* `POST /api/auth/verify-code` ‚Üí set JWT cookie.
* `POST /api/uploads` ‚Üí upload file, extract text, return cleaned text.
* `POST /api/generate` ‚Üí body: `{ company, role, jobUrl?, resumeText }`

  * If `jobUrl` present, **scrape** main content server-side with `got` + `node-html-parser` (handle CORS/timeouts gracefully).
  * Build **prompt** (see ¬ß7) ‚Üí call LLM (gpt-4o-mini or cheaper instruct) to produce:

    * `tailoredResumeMd`
    * `coverLetterMd`
  * Save record, decrement credits (or check subscription).
* `GET /api/history` ‚Üí list last 20 generations.
* `GET /api/generation/:id` ‚Üí fetch one.
* `POST /api/checkout/session` ‚Üí Stripe Checkout session (credits or subscription).

**Rate limiting:** per IP & per user via simple in-memory + DB fallback (e.g., 10 generate/hour on free).

### 7) Prompting strategy (crisp + consistent)

**System prompt (backend constant):**

```
You are a precise career writing assistant that creates ATS-friendly, quantified,
company-specific materials. Output clean Markdown without code fences.
Never invent employment dates or employers; use provided resume only.
If data is missing, ask for it briefly in a single line at the top, prefixed with "MISSING:".
```

**User prompt template (backend):**

```

Target Role: {role}
Company: {company}
Company Info (from job URL if provided, else "N/A"):
{jdExtract}

Candidate Resume (normalized text):
{resumeText}

Constraints:
- Resume: one page, strong section order: Summary, Skills (keywords from JD), Experience (bulleted, quantified),
  Education, Certifications (if any). Avoid tables; use short, dense bullets (max 1‚Äì2 lines each).
- Tailor language to the company's known products, markets, and tone. Include relevant keywords explicitly.
- Cover letter: 3‚Äì5 paragraphs, mention 1‚Äì2 specific company facts from JD or known public info in jdExtract,
  connect 2‚Äì3 quantified achievements, end with a clear CTA.
- Output exactly two headings: "## Tailored Resume" and "## Cover Letter" with content beneath each.
```

**Post-processing:** Split the Markdown into two fields by headings.

### 8) Quality rules

* **ATS-friendly**: plain text/Markdown only, no images/tables. Use standard section headings.
* **Quantification**: Enforce numbers (‚Äúreduced costs by 17%‚Äù, ‚Äúhandled 120+ tickets/week‚Äù).
* **Company alignment**: Pull 3‚Äì8 keywords from JD (store in `jdExtractKeywords` if you want).
* **Guardrails**: If `resumeText` is under 400 chars, return `MISSING:` line prompting user for more data.
